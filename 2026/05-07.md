## 学習テーマ

AWS EBSの概要

## 今日学んだこと

EC2インスタンスで利用するストレージサービスである **EBS（Elastic Block Store）** について学んだ。

EBSは、EC2にネットワーク接続して利用するブロックストレージで、OS・アプリケーション・データの保存に利用される。

---

## EBSとは

EBSは、EC2インスタンスにアタッチして利用するストレージサービス。

主な特徴は以下。

| 特徴 | 内容 |
|---|---|
| ネットワーク接続 | EC2にネットワーク経由で接続されるブロックストレージ |
| データ保持 | EC2を終了してもEBSのデータを保持できる |
| スナップショット | S3にバックアップを保存できる |
| AZに紐づく | 作成されたAZ内で利用する |

---

## EBSの基本特徴

| 項目 | 内容 |
|---|---|
| 可用性 | 高い可用性を持ち、安定稼働を確保できる |
| サイズ | 1GiB〜16TiBなど、用途に応じて容量を選択できる |
| 課金 | 利用時間と容量に応じて課金される |
| 冗長化 | AZ内で複数ハードウェアにレプリカを作成する |
| セキュリティグループ | セキュリティグループ対象外。ポートを閉じてもEBSは利用可能 |

---

## インスタンスストアとEBSの違い

EC2で利用できるストレージには、主に **インスタンスストア** と **EBS** がある。

| 項目 | インスタンスストア | EBS |
|---|---|---|
| 接続方式 | ホストに直接接続された物理ディスク | ネットワーク接続のブロックストレージ |
| 速度 | 高速・広帯域 | 安定した永続ストレージ |
| データ保持 | EC2停止・終了でデータが消える | EC2終了後もデータ保持可能 |
| バックアップ | 基本的に一時利用向け | スナップショットでS3にバックアップ可能 |
| 料金 | インスタンス料金に含まれる | EBS料金が別途発生 |

---

## AWSのストレージサービスの種類

AWSには用途に応じて複数のストレージ形式がある。

| 種類 | 主なサービス | 特徴 |
|---|---|---|
| ブロックストレージ | EBS / インスタンスストア | 低レイテンシ・高IOPS。OSやDBなどに向いている |
| オブジェクトストレージ | S3 / Glacier | 高耐久・低コスト。画像、ログ、バックアップなどに向いている |
| ファイルストレージ | EFS | 複数EC2から同時に共有できるファイルストレージ |

---

## EBSボリュームタイプ

EBSはワークロードに応じて、SSD系・HDD系のボリュームタイプを選択できる。

| ボリュームタイプ | 種類 | 用途 |
|---|---|---|
| 汎用SSD gp3/gp2 | SSD | 一般的なアプリケーション、小〜中規模DB、開発・テスト環境 |
| プロビジョンドIOPS io1/io2 | SSD | ミッションクリティカルDB、高IOPSが必要なワークロード |
| io2 Block Express | SSD | 超高性能DB、高頻度トランザクション、大規模ワークロード |
| スループット最適化HDD st1 | HDD | ビッグデータ、DWH、大規模ログ処理 |
| コールドHDD sc1 | HDD | アクセス頻度が低いデータ、バックアップ、アーカイブ |
| マグネティック legacy | 旧世代 | 旧世代ボリューム。新規利用は非推奨 |

---

## 1ボリューム = 1インスタンスの原則

通常、1つのEBSボリュームは1つのEC2インスタンスにのみアタッチする。

### 原則

| 内容 | 説明 |
|---|---|
| 1ボリューム = 1インスタンス | 1つのEBSボリュームは同時に1つのEC2にアタッチ |
| 複数接続 | 1つのEC2に複数のEBSボリュームを接続することは可能 |
| AZスコープ | EBSは作成されたAZにスコープされる |

### 例外

`io1` や `io2` では、マルチアタッチ機能により、同一AZ内の複数インスタンスで共有できる場合がある。

ただし、複数インスタンスで共有する場合は、データ整合性を守るためにロック制御が必要。

---

## EBSは他AZへ直接アタッチできない

EBSは作成されたAZにスコープされるため、別AZのEC2へ直接アタッチすることはできない。

例：

| EBSの作成AZ | アタッチ可能なEC2 |
|---|---|
| ap-northeast-1a | 同じAZのEC2 |
| ap-northeast-1c | 同じAZのEC2 |
| ap-northeast-1d | 同じAZのEC2 |

別AZで利用したい場合は、スナップショットを作成し、別AZに復元して利用する。

---

## 同一AZ内での付け替え

同じAZ内であれば、EBSボリュームを別のEC2インスタンスに付け替えることができる。

### 付け替え手順

1. ファイルシステムの整合性を確保する
2. 必要に応じてアプリケーションを停止する
3. `unmount` する
4. EC2インスタンスを停止する
5. EBSボリュームをDetachする
6. 別のEC2インスタンスにAttachする
7. 新しいインスタンスでマウントする

### ポイント

- 同一AZ内であれば付け替え可能
- 停止中でもEBSのデータは保持される
- 付け替え前にファイルシステムの整合性を確保することが重要

---

## EBSの暗号化

EBSは、KMSのCMKを利用して暗号化できる。

### 暗号化の対象

| 対象 | 内容 |
|---|---|
| 保存データ | ボリューム内の保存データ |
| 転送データ | ボリュームとインスタンス間の転送データ |
| スナップショット | 暗号化されたボリュームから作成したスナップショット |
| 復元ボリューム | 暗号化スナップショットから作成したボリューム |

### ポイント

- ボリューム作成時に暗号化を有効化できる
- スナップショット作成時にも暗号化される
- デフォルト暗号化を有効にすると、運用を簡素化できる
- 暗号化されたボリュームから作成したスナップショットも暗号化される

---

## EBSスナップショット

EBSはスナップショットを使ってバックアップを取得できる。

スナップショットはAWS管理のS3バケットに保存される。

### スナップショットの特徴

| 特徴 | 内容 |
|---|---|
| 増分バックアップ | 2世代目以降は差分データのみ保存 |
| 圧縮保存 | ブロックレベルで圧縮して保管 |
| 利用継続 | スナップショット作成中もEBSは利用可能 |
| 復元可能 | 別AZや別リージョンで復元可能 |

---

## スナップショットの管理

スナップショットは、整合性を確保しながら管理することが重要。

### 整合性の確保

| 方法 | 内容 |
|---|---|
| アプリ停止 | データ書き込み中の場合、アプリケーションを停止する |
| ファイルシステムのアンマウント | スナップショット作成前にアンマウントする |
| バックアップ機能の活用 | バックアップソフトウェアやエージェントを活用する |

### 自動化

| 方法 | 内容 |
|---|---|
| DLM | Amazon Data Lifecycle Managerでスナップショット取得を自動化 |
| CLI / SDK | AWS CLIやSDKで運用を自動化 |
| タグ管理 | タグを使ってスナップショットの分類・管理を簡素化 |

---

## スナップショット共有

スナップショットは権限を変更することで、他のAWSアカウントと共有できる。

### アカウント間共有

| 項目 | 内容 |
|---|---|
| 権限設定 | スナップショットのアクセス権を変更して特定アカウントと共有 |
| 暗号化Snapshot | KMSキーのキーポリシーや共有設定が必要 |
| パブリック共有 | パブリックまたは特定アカウントに限定共有可能 |

### リージョン間コピー

スナップショットは別リージョンへコピーできる。

| メリット | 内容 |
|---|---|
| 災害対策 DR | 異なるリージョンにバックアップを保持できる |
| 移行・展開 | アプリケーションを別リージョンへ迅速に展開できる |
| コスト最適化 | 必要に応じてコピーし、コストを抑制できる |

---

## EBSボリュームの削除

EBS-backed AMIのEC2インスタンスでは、ルートボリュームにEBSが利用される。

デフォルトでは、EC2インスタンスを削除すると、EBSボリュームも削除される場合がある。

### DeleteOnTermination

| 設定 | 内容 |
|---|---|
| 有効 | EC2インスタンス削除時にEBSも削除される |
| 無効 | EC2インスタンス削除後もEBSボリュームを保持できる |

データを保持したい場合は、`DeleteOnTermination` を無効化する必要がある。

---

## EBSのステータス

EBSボリュームの状態は、主に4つのカテゴリで管理・監視する。

| ステータス | 意味 | 対応 |
|---|---|---|
| OK | 正常に動作している | 監視継続 |
| Warning | 性能低下や一部問題の兆候 | 監視・計画対応 |
| Impaired | 性能や可用性に影響あり | 調査・復旧 |
| Insufficient-data | データ不足で評価不可 | 再評価・確認 |

---

## EBSのRAID構成

EBSでは、パフォーマンス向上や冗長化を目的としてRAID構成を組むことがある。

主に利用されるのは、RAID0とRAID1。

### RAID0：ストライピング

| 項目 | 内容 |
|---|---|
| 目的 | パフォーマンス向上 |
| 仕組み | 複数ボリュームを1つのボリュームのように扱い、データを分散して書き込む |
| メリット | スループットやIOPSを向上できる |
| デメリット | 冗長性なし。1つ壊れるとデータ損失リスクが高い |
| 用途 | 一時データ処理、再生成可能なデータ処理 |

### RAID1：ミラーリング

| 項目 | 内容 |
|---|---|
| 目的 | 冗長性向上・可用性確保 |
| 仕組み | 複数ボリュームに同じデータを書き込む |
| メリット | 片方が壊れてもデータを保持できる |
| デメリット | 書き込み性能が低下する場合がある |
| 用途 | 重要データ、高可用性が必要なワークロード |

EBSは標準で内部的にレプリケートされているため、RAID1の必要性は低下している。

---

## 今日のポイント

- EBSはEC2にアタッチして使うブロックストレージサービス。
- EC2を終了しても、設定によってEBSのデータは保持できる。
- EBSは作成されたAZに紐づくため、他AZのEC2へ直接アタッチできない。
- 同一AZ内であれば、EBSを別のEC2へ付け替えることが可能。
- EBSはスナップショットでバックアップできる。
- スナップショットは増分バックアップで保存される。
- スナップショットは別アカウントや別リージョンへ共有・コピーできる。
- EBSはKMSを使って暗号化できる。
- EBSにはSSD系・HDD系など複数のボリュームタイプがあり、用途に応じて選択する。
- RAID0は性能向上、RAID1は冗長化を目的とする。

---

## 自分用メモ

EBSを理解するときは、以下の観点で整理すると分かりやすい。

1. EBSはEC2用のブロックストレージ
2. インスタンスストアと違い、データを保持できる
3. EBSはAZ単位で管理される
4. 別AZで使う場合はスナップショットから復元する
5. バックアップはスナップショットで取得する
6. スナップショットは共有・リージョンコピーができる
7. 暗号化にはKMSのCMKを利用する
8. ボリュームタイプは性能・コスト・用途に応じて選ぶ
9. 削除時にデータを残したい場合はDeleteOnTerminationに注意する
10. EBSの状態はOK / Warning / Impaired / Insufficient-dataで確認する
